# 改动说明规范

## 禁止的行为

### 不要进行总结
- 禁止在对话结尾进行内容总结
- 禁止使用"总结"、"综上所述"、"总的来说"等总结性开头
- 禁止重复已经说过的内容作为总结

### 错误示例
```
我已经为你创建了文档。

## 总结
这份文档包含了以下内容：
1. 项目基本信息
2. 系统架构设计
3. 开发路线图
...
```

## 必须的行为

### 明确列出改动
当进行了文件操作或代码修改时，必须说明：

1. **新增了什么**
   - 新创建的文件及其用途
   - 新增的功能或代码块
   - 新添加的配置项

2. **修改了什么**
   - 哪些文件被修改
   - 具体修改了哪些行或代码段
   - 修改的原因

3. **删除了什么**
   - 删除的文件或代码
   - 删除的原因

### 正确示例
```
已完成以下改动：

1. 创建文件：`.cursor/rules/communication-style.mdc`
   - 定义了沟通风格规范
   - 禁止emoji和空洞表述
   - 设置为全局生效

2. 修改文件：`项目需求文档.md`
   - 第45-67行：移除了所有emoji符号
   - 第120行：将"太棒了"改为具体的技术分析
   - 第503行：删除了祝福语，改为具体行动建议
```

## 格式要求

### 文件操作说明格式
```
已完成以下操作：

[创建] 文件路径
- 说明1
- 说明2

[修改] 文件路径
- 第X行：具体改动
- 第Y-Z行：具体改动

[删除] 文件路径
- 删除原因
```

### 代码修改说明格式
使用以下结构：
- 文件位置（精确到行号）
- 修改前的代码（如有必要）
- 修改后的代码
- 修改原因

## 适用场景

本规则适用于所有涉及以下操作的回复：
- 创建或修改文件
- 编写或修改代码
- 调整配置
- 重构代码
- 修复bug

## 例外情况

以下场景可以省略详细改动说明：
- 仅回答问题，未进行任何文件操作
- 用户明确要求概述而非详细改动
- 改动过于微小（如修正单个拼写错误）

## 与对话流程的关系

- 先执行操作
- 再说明改动
- 直接结束，不添加总结段落
- 如需后续建议，以"下一步选项"形式呈现

## 增量写入时的说明规范

当使用增量写入策略时，每次写入后的说明应**极度简洁**：

### 正确示例（简洁）
```
[创建] 文档/数据采集方案.md
- 已写入：第1章节（约100行）
- 进度：100/400行（25%）
```

### 错误示例（冗长，会导致超时）
```
我已经为你创建了文档并开始撰写第一部分。
第一章节包含了以下内容：
1. 数据采集基础设定
   - 数据源配置
   - 采集频率设定
   ...（大量内容展开）
现在继续撰写第二章节...
```

### 核心要求
- 每次写入后只说明：文件路径、写入内容、进度
- 不展开解释写入了什么具体内容
- 不在对话中重复文件中的内容
- 立即进行下一次写入，不等待用户回复

## 允许展示的额外内容

在说明改动后，允许（但非必须）展示以下内容：

### 1. 下一步建议
以清单或选项形式提供可操作的后续步骤：

```
下一步选项：
- 选项A：创建数据采集方案文档
- 选项B：搭建信息监测系统框架
- 选项C：配置数据源连接
```

### 2. 待办项（TODO）状态
当存在待办项时，展示进度和未完成项的具体信息：

```
待办项进度：共10项，已完成9项，剩余1项

未完成：
- [待办] 优化数据采集模块的性能问题
```

或简洁格式：
```
待办项：10项中剩余1项未完成
- 优化数据采集模块的性能问题
```

### 注意事项
- 下一步建议应具体、可执行，避免空泛描述
- 待办项状态应包含数量统计和具体未完成内容
- 这些内容是对改动说明的补充，不是总结性重复
