# 增量文档写入策略

## 核心原则

当创建或修改较大文档时，采用增量写入策略，避免长时间无反馈导致响应超时。

### 优先创建文件
- 当用户要求撰写内容或指定保存文件夹时，**立即创建目标文件**
- **禁止**在对话中先规划、思考、长篇输出内容
- **禁止**在对话中展示将要写入的内容
- 第一优先级：创建文件并写入，而非对话输出
- 先创建文件并写入第一部分内容，再继续补充

### 及时写入原则
- 每撰写约 100 行内容就进行一次写入
- 不要等到积累几百行、输出变慢时才写入
- **绝对禁止**在对话中长篇输出内容而不执行文件写入操作
- 保持响应流畅，避免用户长时间等待和超时错误

**重要**：增量写入必须遵循 [file-operation-strategy.mdc](mdc:file-operation-strategy.mdc) 规范：
- 所有分批写入都在**同一个文件**上操作
- 严格禁止创建多个版本文件（如：文档_v1.md、文档_v2.md、文档_草稿.md）
- 使用用户明确指定的文件路径，不自行创建新文件名

## 触发条件

以下情况必须使用增量写入：

### 1. 创建新文档
- 任何预计内容超过 100 行的文档
- 预计内容超过 3000 字
- 包含多个主要章节（2个以上）
- 用户要求撰写内容并指定保存位置时

### 2. 大幅修改现有文档
- 修改内容超过 50 行
- 重构整个文档结构
- 添加大量新章节

## 增量写入策略

### 分批写入原则
1. **立即创建文件**：用户提出需求后，立即创建文件并写入第一部分（约100行或第一章节）
2. **持续增量写入**：每撰写约100行就追加写入一次
3. **最终补充**：完成剩余内容和收尾

### 写入时机
- 达到 100 行内容时，立即写入
- 完成一个完整逻辑章节时（即使未到100行）
- 感知到输出开始变慢时，立即写入当前已完成部分
- 不要等到积累大量内容才一次性写入

### 每批次大小
- **标准批次：约 100 行**
- 或每次写入一个完整的逻辑章节（如果章节小于 100 行）
- 避免在句子或段落中间中断
- 严禁积累几百行才写入（会导致输出变慢、响应超时）

### 写入间隔说明
每次写入后必须说明：
- 当前已完成的部分
- 即将写入的部分
- 整体进度（如：第 2/5 部分）

## 文件路径使用规范

### 严格使用用户指定的路径
- 用户会明确指定文件路径（如：`docs/数据采集方案.md`）
- 必须使用该路径，不得修改文件名
- 不得在用户指定路径外创建其他版本文件

### 禁止的行为
```
错误示例（用户要求创建 数据采集方案.md）：
1. 首次写入创建：数据采集方案_草稿.md
2. 第二次写入创建：数据采集方案_v1.md
3. 最终写入创建：数据采集方案.md

这种做法违反了 file-operation-strategy.mdc 规范！
```

### 正确的行为
```
正确示例（用户要求创建 数据采集方案.md）：
1. 首次写入：数据采集方案.md（第1章节）
2. 第二次写入：数据采集方案.md（追加第2章节）
3. 第三次写入：数据采集方案.md（追加第3章节）
...
始终操作同一个文件
```

## 操作方式

### 使用 write 工具的增量模式

#### 第一次写入
```
使用 write 创建文件（用户指定的路径），包含：
- 文档标题
- 目录结构（如适用）
- 第一个主要章节的完整内容
```

#### 后续追加写入（方式一：推荐）
```
1. 使用 read_file 读取当前内容（从用户指定的文件）
2. 使用 write 覆盖同一文件，内容 = 原内容 + 新章节
3. 不创建新文件，不修改文件名
```

#### 后续追加写入（方式二）
```
使用 search_replace 在文档末尾前插入新内容（同一文件）
```

**关键**：无论哪种方式，都始终操作用户指定的同一个文件。

### 示例流程

用户要求：在 `docs/数据采集方案.md` 创建包含 5 个章节的文档（预计 400 行）

**步骤 1：立即创建文件**
```
[创建] docs/数据采集方案.md
- 已写入：标题、目录、第1章节（约 100 行）
- 即将写入：第2章节
- 进度：100/400 行（25%）
- 操作：write("docs/数据采集方案.md", 第1部分内容)
```

**步骤 2：追加100行**
```
[追加] docs/数据采集方案.md
- 已写入：第2章节（约 100 行）
- 即将写入：第3章节
- 进度：200/400 行（50%）
- 操作：read_file + write（同一文件）
```

**步骤 3：追加100行**
```
[追加] docs/数据采集方案.md
- 已写入：第3章节（约 100 行）
- 即将写入：第4章节
- 进度：300/400 行（75%）
```

**步骤 4：完成剩余部分**
```
[完成] docs/数据采集方案.md
- 已写入：第4-5章节（约 100 行）
- 文档总计：400 行
- 文件数量：1 个（不是多个版本文件）
- 写入次数：4 次（每次约 100 行）
```

## 说明格式

每次写入后的说明应包含：

```
[写入进度] 文档名.md

已完成部分：
- 第一章：XXX（30行）
- 第二章：XXX（45行）

即将写入：
- 第三章：XXX（预计40行）

整体进度：2/5 章节完成
```

## 不适用增量写入的场景

以下情况可以一次性写入：
- 小型文档（< 100 行）
- 简单配置文件
- 代码文件（通常较短）
- 用户明确要求一次性完成

## 强制增量写入的场景

以下情况必须使用增量写入：
- 任何预计超过 100 行的文档
- 用户要求撰写详细内容的文档
- 多章节结构的文档
- 感知到输出开始变慢时

## 避免超时的关键原则

### 问题根源
超时错误 "Tool call ended before result was received" 通常因为：
- 在对话中长时间输出内容而未执行文件操作
- 积累大量内容在内存中未及时写入文件
- 先思考、规划很久才开始文件写入

### 解决方案
1. **立即文件操作**：看到撰写需求后，第一件事是创建文件并写入
2. **禁止对话输出内容**：不要在对话中展示将要写入的内容
3. **优先级：文件写入 > 对话说明**
4. **每100行必须写入一次**：不得例外

### 操作顺序
```
错误顺序（会导致超时）：
1. 读取文件
2. 在对话中输出："现在开始撰写..."
3. 在对话中输出大量内容（几百行）
4. [超时，未执行文件写入]

正确顺序：
1. 简短说明："现在创建文件"
2. 立即调用 write 工具写入第一部分（约100行）
3. 说明进度
4. 继续调用 write 写入第二部分（约100行）
5. 重复直到完成
```

## 与用户的交互

### 立即开始创建
当用户要求撰写内容时：
```
现在创建文件并开始写入第一部分（预计总计约 XXX 行，每 100 行写入一次）。
```

**严格禁止的行为**：
- 在对话中长篇输出待写入的内容
- 先规划、思考、展示大纲再创建文件
- 在对话中撰写内容但不执行文件写入操作

**正确做法**：
- 简短说明后立即调用 write 工具创建文件
- 在文件中撰写内容，而非在对话中撰写

### 写入过程中
不等待用户回复，自动连续写入各部分，但每次都说明进度。

### 意外中断处理
如果写入被中断：
```
文档写入未完成，当前进度：3/5 章节。
你希望我：
1. 继续完成剩余部分
2. 暂停，稍后继续
3. 调整内容后再继续
```

## 性能考虑

### 避免过度频繁
- 不要每写 5-10 行就保存一次（太频繁）
- 保持每批次有实质性内容（约 100 行或一个完整章节）

### 批次大小权衡
- **批次太小**（< 50 行）：频繁 I/O，用户体验差
- **批次太大**（> 200 行）：等待时间长，输出变慢，响应超时风险
- **推荐批次**：100 行左右（经验证的最佳平衡点）

### 响应流畅性
- 当感知到输出速度变慢时，立即写入
- 不要让用户长时间看到文字在对话框中缓慢输出
- 及时写入可以释放缓冲区，保持响应流畅

## 与其他规则的关系

- 本规则与 `change-documentation.mdc` 配合：说明每次增量写入的具体改动
- 本规则与 `file-operation-strategy.mdc` 配合：始终修改同一文件，不创建版本副本
- 增量写入过程中不使用 emoji 和空洞表述（遵循 `communication-style.mdc`）

## 检查清单

在创建大型文档前：
- [ ] 是否预计内容较多（> 100行）？
- [ ] 是否已规划好分批次的逻辑？
- [ ] 每个批次是否是完整的逻辑单元？
- [ ] 是否准备好进度说明？
- [ ] 是否确认了用户指定的文件路径？
- [ ] 是否确保所有写入都在同一文件上操作？

## 用户期望

用户希望看到文档逐步生成，而不是长时间等待后突然出现完整文档。增量写入提供：
1. 可见的进度反馈
2. 中途检查和调整的机会
3. 避免响应超时风险
4. 更好的交互体验
